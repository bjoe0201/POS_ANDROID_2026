# 1. 專案現況摘要

`POS_ANDROID_2026` 是一套以 Android 平板為主要使用情境的火鍋店 POS 記帳系統。目前文件顯示版本為 **v1.2.5 / versionCode 13**，整體已具備完整本地端營運流程，包含 PIN 登入、桌號點餐、菜單管理、桌號管理、訂位管理、報表統計、CSV 匯出、資料備份、自動備份、崩潰保護與點餐操作優化。

目前專案已從「基本 POS 功能」進展到「現場可靠性」與「操作效率」階段。近期版本重點包含：

- 避免系統崩潰 / 斷電造成資料遺失。
- 導入自動備份與備份 UI 即時刷新。
- 加強報表視覺化與 CSV 匯出。
- 優化點餐加減數量操作，加入長按連續加減、Popup 氣泡與觸覺回饋。

### 🔐 PIN 登入

- 4 位數字 PIN。
- SHA-256 雜湊儲存。
- 預設 PIN：`1234`。
- 連續錯誤 3 次鎖定 30 秒。

- 觸覺回饋可於設定開關。
- 近期修正：
  - 結帳品項很多時 Dialog 內容可捲動。
  - 數量歸 0 時隱藏氣泡。
  - 菜單卡片從 0 開始也可長按連續累加。

### 📅 訂位管理

- 月曆月覽。
- 日期進入當日訂位列表。
- 新增 / 編輯 / 刪除訂位。
- 營業時間設定。
- 休息時間設定。
- 預設用餐時間設定。
- 月曆每行 chip 數可調整。

### 🥩 菜單與群組管理

- 品項新增 / 編輯 / 刪除。

- 桌號名稱。

- 啟用 / 停用。

- 手動 ZIP 匯出 / 匯入整個 SQLite DB。

- 自動備份保留天數設定。
- Room 使用 `TRUNCATE` journal + `synchronous=FULL`。
- 點餐 / 結帳 / 取消訂單後 flush。
- App 進背景 checkpoint。
- 自動備份完成後 UI 即時刷新。

## 2.2 主要分層

目前主要程式位於：

```text
app/src/main/java/com/pos/app/
```

主要結構：

```text
data/
  datastore/
    SettingsDataStore.kt
  db/
    entity/
    dao/
    AppDatabase.kt
  repository/

di/
  AppModule.kt

ui/
  login/
  order/
  reservation/
  menu/
  table/
  report/
  settings/
  navigation/
  theme/

util/
  BackupManager.kt
  AutoBackupManager.kt
  SoundEffects.kt
```

目前分層方向正確：

- Entity / DAO 集中在 `data/db`。
- Repository 作為資料存取封裝。
- DataStore 集中處理設定。
- ViewModel 透過 Hilt 注入 Repository。
- Compose UI 以功能模組切分。

## 2.4 資料庫觀察

目前主要 Room Entity 包含：

- `MenuGroupEntity.kt`
- `MenuItemEntity.kt`
- `OrderEntity.kt`
- `OrderItemEntity.kt`
- `ReservationEntity.kt`
- `TableEntity.kt`

對應資料表：

| Table | 用途 |
|---|---|
| `menu_groups` | 菜單群組 |
| `menu_items` | 菜單品項 |
| `orders` | 訂單主檔 |
| `order_items` | 訂單品項明細 |
| `tables` | 桌號設定 |
| `reservations` | 訂位資料 |

值得肯定的資料設計：

- `orders.tableName` 採快照設計，避免桌號改名後歷史資料失真。
- `order_items` 保留品項名稱、價格、群組代碼、群組名稱快照。
- 報表使用軟刪除，避免直接破壞歷史資料。
- 已有 Room migration 演進至 DB v4。
- 已針對崩潰風險強化 journal / fsync 策略。

## 2.5 架構待改善點

### 2.5.1 `CLAUDE.md` 的資料庫表格需補齊

`CLAUDE.md` 目前資料庫結構表格未列出 `reservations`，但實際已有 `ReservationEntity.kt`。

建議後續補上，避免文件與實作不同步。

### 2.5.2 Room `exportSchema = false` 不利長期維護

若目前仍為 `exportSchema = false`，建議後續改為輸出 schema，並建立 migration test。

原因：

- 版本增加後 migration 風險提高。
- 可避免上線後使用者資料庫升級失敗。
- 方便追蹤 DB schema 差異。

### 2.5.3 測試仍偏不足

目前專案已進入實務可用階段，但測試覆蓋仍應補強：

- 訂單加減數量。
- 結帳金額。
- 報表統計。
- CSV 匯出。
- 備份還原。
- DataStore 預設值與設定保存。

### 2.5.4 Release tag 與 main 需保持一致

近期 Git 紀錄顯示 `v1.2.5` tag 之後仍有 hotfix：

```text
39cf88b fix(order): hide bubble when qty reaches 0, enable long-press repeat on card for first add
6905f05 fix(order): make checkout dialog items list scrollable to prevent total/remark from being clipped
30718b0 (tag: v1.2.5) feat(order): long-press repeatable +/- with bubble popup, haptic toggle (v1.2.5)
```

若 APK 是用最新 `main` 打包，但 tag 停在較舊 commit，會造成 Release asset 與 source tag 不一致。

建議後續以 `v1.2.6` hotfix 正式整理，而不是反覆移動既有 tag。

# 4. 目前主要風險與注意事項

## 4.1 Release 一致性風險

目前 `main` 已有 `v1.2.5` tag 後的修正，建議後續正式切 `v1.2.6`，避免 tag、APK、原始碼不一致。

建議：

- 不要移動已公開 tag。
- 新增 `v1.2.6` hotfix。
- 將兩個 hotfix 寫進 `CHANGELOG.md`。
- 確保 Release APK 由同一 commit/tag 建置。

---

## 4.2 資料安全仍是最高優先

目前已做很多保護，但 POS 使用情境下，仍建議加強：

- 備份失敗提示。
- 最近備份健康狀態。
- 備份檔可讀性檢查。
- 還原前 schema / version 檢查。
- 還原前顯示備份摘要。

---

## 4.3 文件可能逐步落後

目前文件已相對完整，但仍有：

- `CLAUDE.md` 資料庫表格缺少 `reservations`。
- `PLANS/plan-hotPotPosApp.prompt.md` 是早期完整規格，與現況已有差異。
- Release Notes 曾被手動覆蓋，建議建立固定發版 checklist。

---

## 4.4 報表模型會受未來商業規則影響

若後續新增：

- 付款方式。
- 折扣。
- 招待。
- 服務費。
- 退菜。
- 日結。

將影響：

- 訂單總額。
- 報表統計。
- CSV 匯出。
- 歷史訂單快照。
- DB migration。

建議在新增前先設計訂單金額模型。

---

## 4.5 硬體與同步功能風險高

印表機、多裝置同步、雲端備份等功能商業價值高，但風險也高：

- 硬體相容性。
- 藍牙 / USB / 網路權限。
- 中文列印。
- 離線同步衝突。
- 資料一致性。
- 後端維護成本。

建議放到中長期規劃。

# 9. 結論

`POS_ANDROID_2026` 目前已經是一套功能完整、接近現場可用的本地端 POS App。它的核心優勢是：

- 點餐結帳流程完整。
- 菜單、桌號、訂位、報表都有基本管理能力。
- 資料安全意識強，已加入崩潰保護與自動備份。
- 文件與版本紀錄相對完整。
- 最近已開始針對現場快速操作體驗做優化。

接下來建議依序處理：

1. **先整理版本與 Release 一致性**：建議切 `v1.2.6` hotfix。
2. **補強備份健康狀態與測試**：提高 POS 核心信任度。
3. **導入付款方式與日結**：讓系統更貼近日常收銀對帳。
4. **加入折扣 / 招待 / 服務費**：補足餐飲現場高頻需求。
5. **再做桌況、訂位整合、印表機、多裝置同步**：逐步往正式商用 POS 發展。

整體而言，此專案的方向正確，下一階段重點應放在：

- 發版流程穩定化。
- 資料安全可驗證化。
- 結帳 / 對帳商業規則完整化。
- 前台營運流程整合化。
